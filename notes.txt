4. ARCHITECTURAL SUGGESTIONS FOR V2
====================================
   
   a) Event-driven architecture
      Para escalabilidad futura (1000+ ordenes/min):
      - Message queue: AWS SQS o RabbitMQ
      - Event types: OrderCreated, OrderPaid, OrderDelivered
      - Microservices: Auth, Orders, Delivery, Analytics
      - Pros: Scalable, decoupled, resilient
      - Cons: More complex ops

   b) Caching strategy
      Actualmente: Simple in-memory cache
      Mejorar a: Redis (persistence, shared across instances)
      Casos: Product list, security logs, token blacklist
      TTL: Products 1h, logs 24h, tokens 30d

   c) Search functionality
      Si muchas órdenes (10k+):
      - Elasticsearch/OpenSearch en lugar de DB queries
      - Indexing: order code, customer name, status, date
      - Faceted search: by status, date range, customer

   d) Background jobs
      Actualmente: Todo síncrono
      Mejorar a: Job queue (Bull, Quirrel, etc)
      Casos: Email send, report generation, cleanup
      Ventajas: No bloquea main thread, retry logic, scheduling

   e) Database optimization
      - Connection pooling: Vercel + Supabase pooler
      - Query optimization: EXPLAIN ANALYZE
      - Materialized views: Pre-computed dashboards
      - Partitioning: Orders by date (historical archival)

   f) API versioning
      Prepararse para:
      - /api/v1/:endpoint (current)
      - /api/v2/:endpoint (future, breaking changes)
      - Deprecation headers: Sunset, Link

   g) GraphQL (Optional future)
      Ventajas: Flexible queries, single endpoint, strong typing
      Desventajas: Learning curve, caching complexity
      Si implementar: Apollo Server con Next.js
