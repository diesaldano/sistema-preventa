3. STAFF

AUTENTICACIÓN (JWT + DB)
========================

R4.1: POST /api/auth/login
      - Input: email + password
      - Validar contra users table (password_hash con bcrypt)
      - Si OK: generar JWT (signado con secret)
      - JWT contiene: userId, role, email, expiresAt (15 min)
      - Retornar: { token, refreshToken, expiresIn }

R4.2: POST /api/auth/refresh
      - Input: refreshToken
      - Validar que NO está en blacklist de tokens
      - Validar firma + expiración (7 días)
      - Si OK: generar nuevo JWT
      - Retornar: { token, expiresIn }

R4.3: POST /api/auth/logout
      - Input: JWT + refreshToken
      - Agregar ambos a tabla blacklist (TimestampToken)
      - Retornar: { success: true }
      - Cliente borra JWT localmente

MIDDLEWARE & ROUTE PROTECTION (R4.x)
====================================

→ Middleware: verifyJWT(req)
   1. Extraer JWT de header Authorization: Bearer <token>
   2. Validar firma + no expirado
   3. Extraer payload: userId, role
   4. Verificar userId NOT en blacklist
   5. Retornar userId + role al handler

→ Middleware: requireRole(roles: [])
   1. Aplicar verifyJWT primero
   2. Validar que usuario.role esté en roles permitidos
   3. Si no: retornar 403 Forbidden
   4. Si sí: continuar

IMPLEMENTACIÓN:
  - /api/orders [GET] → requireRole(['admin'])
  - /api/orders [POST] → PÚBLICO (sin middleware)
  - /api/orders/[code]/validate [POST] → requireRole(['admin'])
  - /api/orders/[code]/deliver [POST] → requireRole(['staff'])
  - /admin/* → requireRole(['admin'])
  - /despacho/* → requireRole(['staff', 'admin'])
  - /api/users/* → requireRole(['admin'])

==========================

  ✓ Costo: 12 (≈250ms por hash, good balance)

  ✓ Access token: short-lived (15 min)

  ✓ Refresh token SOLO via httpOnly cookie (si posible)

RATE LIMITING (Global + Por Usuario):
  ✓ PHASE 1 cubre por IP/email en compra
  ✓ PHASE 4 agrega: /api/auth/login limitado (5 intentos / 15 min por IP)
  ✓ Previene fuerza bruta contra admin/staff

DATABASE SECURITY:
  ✓ Supabase RLS (Row-Level Security) para users table
  ✓ Admin solo ve su propio usuario + staff creados
  ✓ Backups automáticas each 24h (Supabase)
  ✓ Encrypted passwords at rest

HTTPS & VERCEL:
  ✓ Vercel SIEMPRE HTTPS (no permite HTTP)
  ✓ Headers seguridad: HSTS, CSP, X-Frame-Options
  ✓ Rate limit global en edge (Vercel+Cloudflare)
  ✓ DDoS protection built-in

ESCALABILIDAD (500-1000):
  ✑ JWT es stateless → no necesita sessions en DB
  ✑ Si 1000 usuarios concurrentes:
    • Token verification: O(1) via JWT signature
    • Blacklist check: O(1) via hash lookup + Redis cache opcional
    • No bottleneck de sesiones

  ✑ Vercel edge functions geo-distributed (baja latencia)
  ✑ Token caching en cliente (localStorage) = menos peticiones


CASOS EDGE & SEGURIDAD
======================

CASO: Admin pierde password
  Solución: POST /api/auth/reset-password (envía email con link temporal)
  - Link valido solo 30 min
  - Reset token almacenado en DB con expiración
  - Invalidar todas las sesiones del admin (logout all)

CASO: Admin cierra sesión en todos lados (device compromised)
  Solución: POST /api/auth/logout-all-devices
  - Agregar TODOS los refresh tokens del usuario a blacklist
  - Fuerza re-login en todos lados

CASO: Token robado mientras se transmite
  Riesgo: HTTPS mitiga 99%, pero JWT en header sigue visibleISO en cliente
  Solución: 
  - Usar httpOnly cookies (if possible) en lugar de localStorage
  - Implementar CSRF protection
  - Corta duración (15 min) limita daño

CASO: Brute force contra /api/auth/login
  Solución: Rate limit por IP (5 intentos / 15 min)
  - Redis cache (opcional) para velocidad
  - Log cada intento en security_logs (PHASE 3.1)
  - Alert si >5 intentos fallidos

CASO: Staff offline durante evento
  Riesgo: Si WiFi cae, staff no puede marcar entregas
  Solución:
  - Offline mode: cachear últimos N códigos en local storage
  - Cuando vuelve conexión: sincronizar automáticamente
  - Validar que código no fue entregado 2x

DEPLOYMENT FLOW
===============

1. Landing site updates:
   git push → GitHub → Vercel (auto build)
   Vercel invoca ISR en /api/events (cache 1h)
   
2. Sistema preventa updates:
   git push → GitHub → Vercel (auto build)
   Supabase sin cambios

3. Event creation (admin):
   Admin agrega evento en BD (Supabase)
   Landing ISR refresh en next request o revalidate manual
   
4. Blog post publish:
   Crear post en BD (Supabase)
   Landing ISR revalidate /blog, /blog/[slug]
